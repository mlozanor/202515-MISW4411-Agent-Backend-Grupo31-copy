Capítulo 5: Soluciones de Google Cloud para la Gestión de Datos

Gestionar datos en producción no es una tarea menor. Las organizaciones orquestan una sinfonía de servicios para construir pipelines de ingestión, pipelines de transformación, pipelines de entrenamiento de Machine Learning (ML), data lakes y data warehouses. Establecer una cultura de datos madura en una organización implica la adopción de prácticas que pueden ser nuevas para los equipos de tecnología y a menudo requiere la incorporación de nuevas tecnologías. Estas tecnologías abarcan desde soluciones propias del proveedor de la nube hasta software con licencia de terceros, pasando por software de código abierto (OSS). Tanto las organizaciones grandes como las start-ups maduras necesitan comprender la arquitectura de datos, el panorama de soluciones y las compensaciones entre costo y rendimiento en relación con la aplicación que se va a ofrecer.
En este capítulo, se proporciona una visión general de las soluciones de gestión de datos disponibles en Google Cloud desde una perspectiva tanto funcional como de capacidad. Seguiremos el ciclo de vida de los datos (data life cycle) cubriendo desde la ingestión hasta el procesamiento y la entrega de datos.
Comprensión del Recorrido de los Datos
El primer paso para manipular datos en la nube es subirlos al entorno cloud. Dependiendo del sistema de origen y el volumen de datos, se pueden adoptar diferentes enfoques. Para cargas pequeñas, puede ser tan simple como utilizar la interfaz de usuario de Google Cloud para subir un archivo a un bucket de almacenamiento. Sin embargo, a medida que los volúmenes de datos crecen, este simple método basado en la Interfaz Gráfica de Usuario (GUI) puede no ser adecuado. Los clientes de Google Cloud pueden transferir datos de dos maneras principales: una transferencia a través de internet o mediante un transfer appliance.
La forma más conveniente y preferida de transferir datos es generalmente a través de internet. Si existen razones por las cuales no es posible realizar la transferencia en línea, como limitaciones de ancho de banda o costo, los clientes pueden optar por una transferencia mediante hardware. El transfer appliance consiste en un servidor de almacenamiento físico que se envía al cliente, se hidrata con los datos corporativos, y luego se devuelve a Google para su subida a la nube.
Estos métodos son ideales para datos estáticos, pero ¿qué sucede si se necesitan actualizaciones frecuentes, como diarias, mensuales o incluso en tiempo real? Ahí es donde entran en juego los pipelines de datos. Un pipeline de ingestión es un proceso en el que un sistema llama datos desde un sistema fuente y los deposita en un sistema de destino. Esto se puede realizar de forma autónoma y a escala a través de APIs como parte de un pipeline de ingestión automatizado.
La forma principal de ingerir datos en Google Cloud al construir pipelines automatizados es con Pub/Sub. Pub/Sub es un servicio de ingestión de datos sin servidor (serverless) y autoescalable que es excelente para ingerir datos de múltiples fuentes y distribuirlos a través de varios servicios. Por ejemplo, Pub/Sub puede llamar APIs de un sistema de gestión de relaciones con clientes (CRM) y de un sistema de marketing para depositar esos datos en un bucket de Cloud Storage para archivo o retención. También puede enviar los datos a un pipeline de transformación donde serán saneados antes de llegar a su destino final: un almacén de datos (data warehouse), donde pueden ser consumidos por otros sistemas o empleados para un propósito empresarial específico.
Las ventajas de Pub/Sub incluyen su alta disponibilidad (HA) a través de regiones, la entrega de mensajes at-least-once (al menos una vez) con soporte para modos push y pull, y su capacidad para manejar datos en tiempo real. Una solución de ingestión de código abierto comúnmente utilizada es Apache Kafka, una plataforma de streaming distribuido que se utiliza para ingerir grandes volúmenes de datos de múltiples sistemas.
Una vez que los datos están en la nube, los servicios de ingestión típicamente depositan los datos en un pipeline de transformación, donde se limpian y estructuran en armonía con el resto de los datos en un almacén de datos. Alternativa o simultáneamente, los datos ingeridos pueden necesitar ser almacenados en un data lake (lago de datos) en su formato crudo para retención o cumplimiento normativo.
Un data lake es un repositorio central para almacenar grandes volúmenes de datos en su formato nativo. Se construyen típicamente sobre servicios de almacenamiento de objetos. Google Cloud Storage (GCS) es el servicio de almacenamiento de objetos de Google Cloud. Una ventaja de GCS es su capacidad para manejar cualquier tipo de archivo, incluyendo datos estructurados, semiestructurados o no estructurados.
Los buckets de almacenamiento de objetos basados en la nube son escalables y rentables. Su tamaño es dinámico, creciendo o disminuyendo según la cantidad de datos. GCS es puramente un servicio de almacenamiento sin un componente de cómputo, lo que lo hace muy rentable para el almacenamiento de datos.
Una vez que los datos son enviados a un pipeline de transformación, se inicia un proceso ETL (Extract, Transform, Load) o ELT (Extract, Load, Transform). Estos acrónimos indican si la transformación se realiza antes o después de depositar los datos en su destino. En Google Cloud, los servicios más comunes para trabajos ETL/ELT son Dataflow y Dataproc. Dataflow es un servicio serverless de Apache Beam que maneja transformaciones por lotes y en tiempo real. Dataproc puede considerarse Apache Spark serverless. Adicionalmente, Cloud Composer (basado en Apache Airflow) es un servicio completamente gestionado para la orquestación de pipelines.
Una vez que los datos son limpiados, se depositan en uno o varios sistemas para su consumo. Si estos datos son críticos para las operaciones o son externos, pueden almacenarse en una base de datos (database). Si los datos se utilizan para análisis, visualización o ciencia de datos, pueden ir a un data warehouse.
Bases de Datos
Una base de datos es una colección de datos organizada y estructurada para ser fácilmente accesible y buscable por otros sistemas.
• Bases de Datos Relacionales (SQL): Almacenan datos en tablas con columnas y filas. Son ideales para datos sensibles y que requieren una precisión constante, como información bancaria. Deben ser ACID compliant (Atomicidad, Consistencia, Aislamiento y Durabilidad), propiedades que garantizan que las transacciones sean procesadas de manera confiable. Una limitación común es que normalmente solo pueden escalar verticalmente, añadiendo más cómputo y almacenamiento a una base de datos existente.
    ◦ Cloud SQL es el servicio de base de datos relacional completamente gestionado de Google Cloud que soporta MySQL, PostgreSQL y Microsoft SQL Server. Es adecuado para bases de datos relacionales regionales.
    ◦ Spanner es un servicio de base de datos relacional distribuida y completamente gestionada que puede escalar horizontalmente (across multiple nodes) manteniendo la compatibilidad ACID y la replicación automática y síncrona. Es óptimo para datos a escala global que necesitan mantener una fuerte consistencia, como información de vuelos o aplicaciones bancarias globales.
• Bases de Datos No Relacionales (NoSQL): Utilizan diferentes formas de organizar los datos, como pares clave-valor, documentos o gráficos.
    ◦ Memorystore es un servicio de almacenamiento en memoria completamente gestionado para cargas de trabajo Redis y Memcached. Es de baja latencia y altamente escalable, ideal para el almacenamiento en caché de datos que necesitan ser accedidos con frecuencia, como perfiles de usuario o catálogos de productos.
    ◦ Firestore es una base de datos de documentos NoSQL completamente gestionada y autoescalable, ideal para manejar la sincronización de datos en tiempo real en aplicaciones móviles y web, como aplicaciones de chat. Es serverless, lo que le permite escalar según la demanda.
    ◦ Bigtable es un servicio de base de datos NoSQL de columna ancha gestionada. Es capaz de manejar datos estructurados y semi-estructurados a escala de petabytes. Ofrece una latencia de sub-milisegundos y una disponibilidad del 99.999%, lo que lo hace ideal para cargas de trabajo analíticas a gran escala y casos de uso de Internet de las Cosas (IoT).
Almacenes de Datos (Data Warehouses) y BI
Un data warehouse es un repositorio central de datos (idealmente limpios) utilizado para análisis, informes y casos de uso de ML. Está diseñado para ser altamente escalable y para realizar consultas de manera rápida y eficiente.
• BigQuery es el servicio de data warehouse de Google Cloud. Cuenta con una arquitectura desacoplada que permite que el almacenamiento y el cómputo se escalen dinámicamente según la necesidad, lo que lo hace muy performante y rentable. BigQuery también tiene capacidades integradas de ML (BigQuery ML), lo que permite ejecutar predicciones de modelos directamente sobre los datos. Es un servicio first-party (propio) de Google Cloud y es adecuado para almacenar grandes cantidades de datos para análisis histórico y entrenamiento de modelos de ML.
• Looker es la plataforma de Inteligencia de Negocios (BI) de Google Cloud. Se integra con un data warehouse y proporciona una capa de visualización para que los equipos y clientes puedan auto-servirse datos de la plataforma. Permite la construcción y programación de informes y cuadros de mando (dashboards) que se pueden utilizar interna o externamente (incrustados en aplicaciones).
